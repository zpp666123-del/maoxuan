## 1. 产品摘要

### 1.1 产品定义

本产品以项目方选定版本的毛泽东著作为正文主轴。用户阅读到某个段落、点击人物地点、选中文字或提出局部问题时，系统从预先整理的文章历史情境包中检索材料，再由内置提示词和工作流把结果流式填入固定界面的三个核心槽位：

1. **历史背景**：事情为何发展到这里；
2. **当时处境**：站在写作时点，眼前必须解决什么；
3. **思想与推理**：正文怎样判断、怎样推理、导向什么方法。

MVP 只保留固定三槽位方案；空间、事件、人物和选择关系统一进入三个固定槽位，由真实大模型实时生成相关流式输出。

### 1.2 核心创新

产品的创新不在“把资料变成电子文本”，而在以下两层：

- **历史情境加工层**：把文章、事件、人物、地点、力量、约束、争论、选项和思想推理重新组织成可复用的 ContextPack。
- **AI 编排层**：以固定槽位、内置 Prompt、声明式 Workflow、时间边界和质量门禁，决定用户此刻看到什么，而不是让模型自由生成一篇答案。

### 1.3 产品承诺

用户不需要先读大量通史，也不必离开原文查百科。系统应让用户在当前阅读位置快速获得足够但不喧宾夺主的历史上下文，并保留继续深挖的入口。

## 2. 背景与问题

### 2.1 现状

现有数字化产品通常已经提供正文、检索、馆藏陈列、全文数据库或通用 AI 问答，但仍存在四个缺口：

- 文本与历史现场分离，读者不知道文章为何此时出现；
- 背景介绍常是静态长文，与当前读到的段落没有联动；
- AI 问答容易脱离选定版本、混用后见之明或重复泛泛概括；
- 人物、地点、事件和争论常被做成孤立展项，没有形成围绕文章的统一阅读过程。

### 2.2 用户痛点

- 原文密度高、时代距离远，进入成本高；
- 知道字面意思，但不知道文章具体在回应谁、解决什么；
- 背景资料很多，却难以判断哪些与当前段落真正相关；
- 普通摘要减少了阅读，不一定增加理解；
- 通用聊天问答缺乏稳定结构，每次回答质量和深度不同；
- 历史结果容易反向覆盖当时处境，形成后见之明。

### 2.3 要解决的核心问题

> 当用户读到文章中的一个论证节点时，如何在不打断阅读的前提下，让他理解这段文字的前因、当下压力、人物关系、现实争论和思想推理？

## 3. 目标与非目标

### 3.1 MVP 目标

- 建立一套稳定、通用的历史情境阅读界面；
- 建立文章、锚点、ContextPack、交互和生成记录的数据标准；
- 形成可版本化的 Prompt Registry 与 Workflow Engine；
- 实现唯一 `framework_stream` 路径；
- 接入真实大模型，实时生成背景、处境、思想三槽位流式输出；
- 建立内容编辑、审核、生成、评测和发布闭环；
- 用一篇文章完成完整纵向切片，并验证可复制性。

### 3.2 非目标

MVP 不承担：

- 全集一次性加工；
- 多版本校勘、异文比较；
- 全量原始扫描影像浏览；
- 无边界的中国近现代史百科；
- 无资料约束的通用政治历史聊天；
- 以 3D、VR 或数字人为核心的展陈；
- 自动替代历史研究者和内容审核者；
- 对所有争议给出单一确定结论。

## 4. 产品原则

1. **原文第一**：AI 解释不能遮蔽、改写或替代正文。
2. **框架先于生成**：产品定义界面和槽位，模型只填内容。
3. **处境先于评价**：先理解当时可见的信息、限制和选择，再看后续结果。
4. **结构化资料先于模型记忆**：具体事实优先来自 ContextPack。
5. **少而有用**：每次只展示与当前锚点最相关的信息。
6. **唯一实时路径**：阅读端只走 `framework_stream`，不保留任何并行替代路径。
7. **轻量可追踪**：所有卡片至少能追溯到正文段落 ID、背景资料 ID 和生成版本。
8. **人工可纠正**：内容团队能够在工作台修订资料、Prompt、Workflow 和发布版本。
9. **后见信息显式隔离**：默认冻结在写作时点；如需后续结果，必须由 Prompt 规则显式标注。
10. **模型可替换**：Prompt、Workflow 和数据协议不依赖单一模型供应商。

## 5. 目标用户与角色

### 5.1 阅读用户

- 对毛泽东著作有兴趣但难以持续进入的普通读者；
- 希望结合历史现场理解文章的深度读者；
- 学生、教师、研究者或内容创作者；
- 更习惯交互式、视觉化和问题驱动阅读的年轻用户。

### 5.2 内容角色

| 角色 | 职责 |
|---|---|
| 内容编辑 | 导入正文、划分锚点、整理情境单元、补充来源 |
| 历史审核者 | 审查时间、人物、事件、处境和后见信息边界 |
| 思想内容审核者 | 审查论证结构、概念含义和文本关联 |
| 发布者 | 运行质量门禁、发布或回滚 ContextPack、Prompt 与 Workflow |
| 管理员 | 权限、模型、Prompt、Workflow、审计和环境配置 |

## 6. 核心用户任务（JTBD）

1. 当我读到一段难以理解的文字时，我想马上知道它之前发生了什么，以免先去读大量外部资料。
2. 当我看到一个判断时，我想站在当时的条件下理解为何会这样判断，而不是只听后来的评价。
3. 当文章涉及地点、路线、力量分布或争论时，我希望这些关系被压缩进当前槽位，而不是跳到另一套展示方案。
4. 当我提出局部问题时，我希望系统仍然围绕当前文章、锚点和固定三槽位回答。
5. 当我想深挖一个词、一句判断或一个人物时，我希望系统只围绕当前文章与选区回答。
6. 当我回到文章继续阅读时，我希望系统记住位置、展开状态和个人标记。
7. 当内容有错误或不清楚时，我希望能反馈到具体卡片和具体事实。

## 7. 核心体验与信息架构

### 7.1 页面骨架

```text
┌──────────────── 文章定位条 ────────────────┐
│ 标题｜时间｜地点｜文体/对象｜文章核心问题 │
├──────────────────┬─────────────────────────┤
│                  │ 历史背景                │
│                  │ 此前怎样走到这里        │
│    原文阅读区     ├─────────────────────────┤
│                  │ 当时处境                │
│ 当前锚点 / 选区   │ 此刻必须解决什么        │
│                  ├─────────────────────────┤
│                  │ 思想与推理              │
│                  │ 如何判断、为何、怎么做  │
└────────────────────────────────────────────┘
```

### 7.2 三个固定槽位

| 槽位 | 必须回答 | 禁止变成 |
|---|---|---|
| 历史背景 | 当前问题形成前最相关的 1–3 个事件或变化 | 通史罗列、思想结论、后续评价 |
| 当时处境 | 写作时点的核心难题、力量、约束、选项、未知 | 背景复述、作者结论复述、结果倒推 |
| 思想与推理 | 本段判断、依据、推理链和行动方法 | 口号解释、人物赞美、与本段无关的概念百科 |

### 7.3 唯一生成框架

阅读端只保留 `framework_stream`：

| 固定框架 | 说明 |
|---|---|
| 原文阅读区 | 只展示选定版本正文，不插入模型改写 |
| 历史背景 | 由真实大模型围绕当前锚点实时生成 |
| 当时处境 | 由真实大模型围绕写作时点限制实时生成 |
| 思想与推理 | 由真实大模型围绕当前段落或选区实时生成 |

不再提供其他并行方案。

## 8. 用户流程

### 8.1 首次进入文章

1. 用户从文章列表或搜索进入文章；
2. 页面加载选定版本正文和文章定位条；
3. 前端创建阅读会话并提交 `anchor_changed` 交互；
4. 后端调用真实大模型，按三槽位实时流式输出；
5. 用户继续点击段落、选中原文或提问，仍走同一条 `framework_stream`。

### 8.2 统一实时生成路径

1. 用户进入文章、点击段落、选择文本或提交局部问题；
2. 前端发送标准交互事件；
3. 编排器检索当前文章、当前锚点和 ContextPack；
4. 真实大模型分别生成背景、处境、思想三个槽位；
5. 服务端通过 SSE 将模型流式输出写入预制框架；
6. 完整槽位对象通过校验后提交为最终 UI 状态；
7. 生成记录入库，保留 Prompt、模型、输入和输出版本。

### 8.3 局部提问

- 默认问题范围为当前文章和锚点；
- 超出馆藏范围的问题应收窄或提示范围，而不是用模型记忆自由扩写；
- 回答必须回填三个固定槽位，不取代阅读页。

## 10. 数据与内容要求

### 10.1 Article

- 唯一文章 ID；
- 选定版本与版本说明；
- 标题、时间、地点、文体、对象、写作目的；
- 稳定段落 ID；
- 锚点列表与摘要；
- 文章级核心问题。

### 10.2 ContextPack

每篇文章至少包含：

- 写作前关键事件；
- 写作时点的核心问题；
- 主要人物与组织状态；
- 现实限制与信息边界；
- 当时争论和可选方向；
- 本文重要概念的局部含义；
- 必要地点与空间关系；
- 锚点到情境单元的明确连接；
- 轻量来源登记；
- 文章日期与后见信息截止规则。

### 10.3 生成记录

每次生成记录必须绑定“文章版本 + 锚点 + ContextPack 版本 + Prompt 版本 + Workflow 版本 + 模型策略 + 输入哈希 + 输出对象”。所有记录可追踪、可审计、可复盘，但阅读端不再依赖预生成内容。

## 11. 非功能需求

### 11.1 性能目标

- 已缓存文章正文首屏 P95 小于 2 秒；
- 交互创建接口 P95 小于 300ms；
- 首个流式语义块 P95 小于 4 秒；
- 三槽位完整生成 P95 小于 15 秒；
- SSE 中断后可查询交互最终状态。

以上为研发目标，可在压测后调整。

### 11.2 可用性与降级

- 模型不可用时仍可完整阅读正文，但槽位必须明确失败；
- 单个槽位失败不阻塞其他槽位；
- SSE 中断支持用交互 ID 恢复或查询最终结果；
- 所有实时结果必须有超时和最大重试次数。

### 11.3 一致性

- 发布内容读取强一致；
- 运行记录、分析事件可最终一致；
- ContextPack 更新后，新生成必须绑定明确版本；
- 客户端使用 `uiVersion` 防止过期流覆盖新交互。

### 11.4 可访问性

- 键盘可操作；
- 流式更新有文字等价说明；
- 颜色不是唯一编码；
- 正文可调字号、行距、列宽；
- 流式更新有适度的 `aria-live`，避免屏幕阅读器重复播报。

## 12. 质量与成功指标

### 12.1 内容质量指标

- Schema 合法率：100%；
- 明确事实的来源覆盖率：100%；
- 时间冻结泄漏率：小于 2%；
- 无依据具体事实率：小于 1%；
- 三槽位语义混淆率：小于 5%；
- 三槽位内容重复率：小于 8%；
- 人工审核平均质量分：不低于 4/5；
- `unavailable` 合理率：宁可缺失，不以猜测填充。

### 12.2 产品指标

首期不以日活为唯一指标，重点验证：

- 文章完成阅读率；
- 锚点深挖率；
- 用户在展开情境后返回原文继续阅读的比例；
- 三槽位生成完成率和失败原因分布；
- 局部问题得到解决的自评比例；
- 同一用户的回访与跨文章迁移；
- “不准确/不相关/重复”反馈率的下降趋势。

### 12.3 技术指标

- 每次交互模型调用数；
- 首卡延迟、完整延迟；
- 校验失败率、修复成功率；
- 单交互 Token 与成本；

## 14. 风险与对策

| 风险 | 表现 | 对策 |
|---|---|---|
| 内容范围膨胀 | 每篇文章演变为一套通史 | 只收录能解释锚点的问题、处境和推理的材料 |
| 模型写成长文 | 三槽位重复、喧宾夺主 | 独立 Prompt、长度限制、差分更新、校验器 |
| 后见之明 | 用后来结果解释当时判断 | 时间冻结字段、资料时间过滤、专门评测用例 |
| AI 具体事实幻觉 | 自造日期、数字、地点 | 只用 ContextPack、sourceIds 必填、具体字段白名单 |
| 滚动抖动 | 卡片频繁刷新 | 锚点而非逐段、450ms 防抖、最小更新 |
| 内容生产成本高 | 一篇文章需大量整理 | AI 只做候选提取；建立模板、批量关联、复用情境单元 |
| 输出脱离框架 | 模型生成长文或聊天答案 | 固定三槽位、格式约束、Schema 校验 |
| 模型供应商锁定 | Prompt 与 API 深度绑定 | 模型网关、结构化输出、供应商适配器 |
| 发布后不可复现 | 模型变化导致答案漂移 | 版本记录、输入哈希、运行记录可重放 |
| 权利与公开边界 | 选定版本版权或资料权利不清 | 资料登记、公开级别、按条目控制可见性和引用长度 |

## 15. 发布验收标准

### 15.1 一篇文章可发布的 Definition of Done

- 正文已按选定版本导入并人工抽查；
- 每段有稳定 ID，锚点覆盖全文且无重叠缺失；
- 每个锚点至少有 1 条有效背景、1 条处境或思想关联；
- 必要人物、组织、事件、地点和争论已登记；
- 文章级时间截止规则已设置；
- 真实大模型环境变量、模型策略和调用权限已配置；
- Schema 与确定性校验 100% 通过；
- Golden Set 通过率达到发布阈值；
- 历史审核与思想内容审核均通过；
- 前端可在 `framework_stream` 路径完整生成三槽位；
- 模型不可用时明确失败并保留正文阅读；
- 反馈、审计、监控和回滚已验证。

### 15.2 MVP 产品验收

- 一个新用户在不阅读说明文档的情况下能完成：进入文章、阅读、理解三槽位、点击段落、选中文本、提交问题、返回原文；
- 三槽位在连续五个锚点切换中无明显重复和错位；
- 任一具体事实可在两次点击内看到轻量来源信息；
- 同一交互在同一版本下能够重放；
- 同一运行记录能看到模型、Prompt、ContextPack 和 sourceIds；
- 故意注入的选中文本指令不能改变系统规则；
- 前端不会因旧 SSE 流覆盖新锚点内容。
