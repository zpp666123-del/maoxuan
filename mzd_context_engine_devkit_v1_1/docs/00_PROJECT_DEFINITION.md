# 00. 项目定义与设计原则

## 0.1 项目名称

**毛泽东著作历史情境化阅读引擎**  
上位项目：**可验证的毛泽东数字思想博物馆**

## 0.2 本项目加工的不是“更多文本”，而是“历史情境层”

数字化第一步解决载体问题：纸书变成可检索文本。第二步要解决理解问题：把文章重新放回其写作时点，整理出文章所面对的事件链、现实限制、力量关系、争论、可选方向和思想推理。

因此产品的最小单位不是一页 PDF，也不是一条知识图谱边，而是：

```text
某个阅读锚点
+ 与它最相关的历史背景
+ 写作时点的现实处境
+ 本段思想与推理
+ 必要时出现的视觉场景
```

## 0.3 两个核心资产

### 资产一：ContextPack

每篇文章预先整理的结构化历史情境包。它决定模型能够使用哪些事实、时间边界、人物关系、约束、争论、选项和概念。

### 资产二：Prompt + Workflow

模型不是自由回答，而是在预先定义的工作流中完成规划、检索、槽位生成、校验、修复和流式提交。提示词、工作流、模型和数据版本共同决定一次输出，可被重放和审计。

## 0.4 产品设计原则

- **原文不动，解释随动**：正文是稳定中心，旁侧内容随锚点和交互变化。
- **界面有限，内容可深**：固定区域少，但每个区域可展开、可追踪、可视觉化。
- **先提供默认理解，再允许主动深挖**：滚动加载审核运行记录，点击/选中触发实时分析。
- **历史边界由数据控制，不只靠提示词提醒**：资料检索必须按文章日期过滤。
- **视觉化是解释工具，不是展览装饰**：没有明确问题就不生成图。
- **资料不足可以留白**：不强迫每个槽位都有完整答案。
- **人工负责发布，AI 负责加速**：AI 可提取、组织、生成候选，不能自行公开发布。

## 0.5 MVP 的明确简化

- 单一选定版本；
- 不要求原始影像；
- 不做异文校勘；
- 来源显示只到段落锚点和背景资料条目；
- 第一篇只做两类动态场景；
- 不一次性搭建完整知识图谱；
- PostgreSQL 可作为第一阶段唯一权威数据源。

## 0.6 长期演进方向

当核心阅读引擎稳定后，可以逐步扩展：

1. 跨文章思想脉络；
2. 地图与战局、交通、组织空间；
3. 人物与争论关系；
4. 后续行动与结果；
5. 版本与原始影像；
6. 教学关卡、历史推演、专题展览；
7. 对外研究 API 与可引用研究路径。
