# 06. ContextPack：文章历史情境加工规范

## 6.1 定义

`ContextPack` 是围绕一篇文章整理的结构化历史情境包。它不是一篇长背景文章，而是一组可被不同锚点、模式和交互组合使用的情境单元。

它回答四个层次：

```text
文章为何出现
→ 当前段落之前发生了什么
→ 写作时点面临什么处境
→ 正文如何形成判断和方法
```

## 6.2 最小组成

```text
Article Metadata
Canonical Paragraphs
Anchors
Context Units
Entities
Places
Sources
Anchor Links
Time Policy
Published GenerationRuns
```

## 6.3 文章元数据

必填：

- `articleId`
- 标题；
- 写作日期与精度；
- 写作地点与精度；
- 文体；
- 主要对象；
- 写作目的；
- 文章级核心问题；
- 选定版本说明；
- 时间冻结截止日。

对于日期或地点不确定的情况，使用 `precision` 和 `confidence`，禁止为了字段完整而虚构精确值。

## 6.4 段落与锚点

### 段落

- 每个自然段有稳定 `paragraphId`；
- 顺序字段独立于 ID；
- 原文保留基本标点和分段；
- 正文修订产生 articleVersion，不覆盖已发布版本。

### 锚点

锚点是一组完整的论证小节，通常 3–10 段。划分依据：

- 问题发生变化；
- 论证阶段变化；
- 对象或场景变化；
- 从事实判断进入方法；
- 从总论进入具体策略。

锚点不得过细，否则滚动抖动；不得过粗，否则情境内容失去针对性。

每个锚点应有：

- 标题；
- 段落范围；
- 本锚点在全文的作用；
- 主问题；
- 可选关键词；
- 默认场景偏好。

## 6.5 ContextUnit 类型

### `event`

对当前文章或锚点有解释价值的事件。字段：时间、地点、参与者、发生了什么、为何与本段相关。

### `actor_state`

某人物或组织在写作时点的身份、位置、目标、资源、态度和关系。只写当前文章需要的状态，不写完整生平。

### `constraint`

限制判断或行动的现实条件：时间、兵力、资源、组织、交通、信息、舆论、国际环境等。

### `debate`

当时存在的不同判断、争论或路线。必须记录各方观点，而不是只写“存在争论”。

### `option`

当时可能采取的方案或方向。选项可以是历史上明确提出的，也可以是基于资料整理出的合理分类；后者必须标为编辑解释。

### `concept`

某概念在本文具体语境中的含义、与其他概念的区别、在本段论证中的作用。避免写成百科定义。

### `place`

地点、路线、控制区或空间关系。必须说明其对当前问题的重要性。

### `statistic`

确有解释价值的数字。必须有来源、时间口径和单位；非必要不用数字装饰。

### `outcome`

文章发表后的行动、结果、修正或争议。默认仅在“前因后果”模式使用。

### `question`

编辑整理的核心问题，可用于锚点标题、引导提问和质量测试。

### `reasoning_clue`

从原文识别出的推理节点：观察、判断、前提、方法、行动原则。它不是替代思想卡，而是生成时的结构化提示。

## 6.6 Unit 通用字段

```json
{
  "id": "cu_constraint_001",
  "type": "constraint",
  "title": "时间窗口有限",
  "summary": "...",
  "details": "...",
  "validFrom": "YYYY-MM-DD",
  "validTo": "YYYY-MM-DD",
  "knownAt": "YYYY-MM-DD",
  "timeRelation": "at_writing",
  "entityIds": [],
  "placeIds": [],
  "sourceIds": ["src_001"],
  "confidence": "high",
  "factInterpretation": "fact",
  "editorialStatus": "approved",
  "visibility": "public"
}
```

`factInterpretation`：

- `fact`：资料直接支持的事实；
- `interpretation`：编辑或研究者解释；
- `inference`：基于多条资料作出的推断；
- `question`：待研究问题。

## 6.7 Anchor Link

ContextUnit 不直接属于某一个锚点，而通过 Link 连接：

```json
{
  "anchorId": "anchor_03",
  "contextUnitId": "cu_constraint_001",
  "role": "explains_situation",
  "relevance": 0.94,
  "priority": 1,
  "slotHints": ["situation"],
  "sceneHints": ["decision"],
  "editorNote": "解释为何本段必须先判断主要矛盾。"
}
```

角色枚举建议：

- `precondition`
- `trigger`
- `explains_background`
- `explains_situation`
- `supports_reasoning`
- `contrasts_reasoning`
- `defines_concept`
- `spatial_context`
- `later_outcome`

## 6.8 来源登记（轻量）

第一阶段不做重型文献学，但每个具体事实必须指向 Source：

- `sourceId`
- 标题；
- 类型；
- 作者/机构；
- 时间；
- 简短定位，如章节、页码或条目；
- 是否可公开；
- 可靠等级；
- 编辑备注。

来源等级建议：

- `A`：选定版本正文、同期正式材料、权威档案；
- `B`：高质量历史研究、正式出版史料；
- `C`：一般研究、回忆、辅助资料；
- `D`：待核线索，不可用于已发布具体事实。

轻量来源抽屉默认只显示标题、定位和用途。原始影像、复杂异文以后追加，不影响现有 ID。

## 6.9 时间政策

每条 Unit 至少明确三种时间之一：

- 事件何时发生；
- 信息何时可能被当事人知道；
- 资料何时形成。

“回到当时”模式使用的是 `knownAt`，不是资料出版时间。后来的研究可以帮助描述当时状态，但必须明确它不是当时已知信息。

## 6.10 内容生产步骤

### 第 1 步：导入与校对正文

- 确认选定版本；
- 导入文本；
- 清理明显格式问题；
- 人工抽查；
- 生成稳定段落 ID。

### 第 2 步：划分锚点

编辑先人工划分，AI 可提供候选边界与理由。发布前必须人工确认。

### 第 3 步：文章级问题卡

先回答：

- 为什么写；
- 写给谁；
- 当时最核心的问题；
- 文章整体如何推进。

这一步决定后续情境整理范围。

### 第 4 步：建立时间骨架

只整理与文章直接相关的 5–20 个前置节点，避免通史化。

### 第 5 步：整理处境矩阵

模板：

| 维度 | 内容 |
|---|---|
| 核心问题 | 此刻必须解决什么 |
| 力量 | 主要人物/组织及状态 |
| 资源 | 可用与缺少什么 |
| 时间 | 截止、窗口、紧迫性 |
| 空间 | 地点、路线、距离、控制区 |
| 信息 | 已知、未知、误判风险 |
| 争论 | 有哪些不同判断 |
| 选项 | 可能的行动方向 |

### 第 6 步：整理思想推理线索

对每个锚点记录：

```text
问题
→ 观察/证据
→ 判断
→ 隐含前提
→ 方法
→ 行动原则
```

这只是生成材料，不直接作为最终卡片。

### 第 7 步：关联锚点

每个 Unit 至少关联一个锚点；通用 Unit 可关联多个锚点，但相关度和角色不同。

### 第 8 步：场景数据

只对真正需要的锚点补充：

- 时间线节点；
- 决策路径节点；
- 地图坐标/路线；
- 关系图节点/边。

### 第 9 步：生成运行记录

按锚点和模式生成三槽位与场景。

### 第 10 步：审核与缺口补充

审核者不仅看答案是否“正确”，还看：

- 是否解释了当前锚点；
- 是否太多或太少；
- 是否把背景和处境混在一起；
- 是否泄漏后见信息；
- 是否缺少关键约束或争论。

## 6.11 一篇文章的最低发布量

参考下限：

- 8–15 个锚点；
- 10–25 个 event；
- 5–15 个 actor_state；
- 8–20 个 constraint；
- 3–10 个 debate；
- 3–10 个 option；
- 5–15 个 concept/reasoning_clue；
- 10–30 个 Source；
- 每锚点至少 3 个高相关 Unit；
- 所有 P0 模式运行记录。

不是数量越多越好。应以覆盖文章关键问题为准。

## 6.12 内容风格

- 使用现代清晰语言解释，但不改写正文；
- 避免崇拜式、贬损式和先验结论；
- 具体、可理解、可连接到原文；
- 区分事实与解释；
- 不用大量抽象标签代替历史处境；
- 有争议时说明争议，而不是用模糊中性语掩盖差异；
- 不把现实历史苦难设计成娱乐式积分或胜负游戏。

## 6.13 可复用性

ContextUnit 可有两级：

- `article_local`：只在本文有效；
- `collection_shared`：可在多篇文章复用，如同一事件、人物状态或地点。

首期以 article_local 为主，等三篇以上文章出现重复后再抽取共享单元，避免过早建模。

## 6.14 内容缺口机制

AI 或审核者可创建 Gap：

```text
缺少某锚点的核心约束
缺少争论对方的资料
时间线存在断点
地点没有可用坐标
某具体数字只有低等级来源
思想卡依赖的推理线索不明确
```

Gap 有优先级、负责人、状态和阻塞的发布范围。资料不足时可以只阻塞一个槽位或场景，不必阻塞全文。

示例与模板见 `examples/` 和 `templates/`。
